# 分布式系统与常用中间件速记

这份文档面向“之前几乎没接触过、但面试前需要快速补齐”的场景。

目标不是让你学成架构师，而是让你达到：

1. 能听懂常见分布式和中间件问题在问什么。
2. 能回答高频八股，不至于一句都说不出来。
3. 能结合你自己的项目，讲到“概念、问题、常见方案、权衡”的程度。

---

## 一、你要掌握到什么程度

先说结论：

1. 不要求你会设计复杂的分布式架构。
2. 不要求你深挖 `Raft`、`Paxos`、`Kafka`、`Redis` 源码。
3. 你最需要的是“能解释、能举例、能说常见处理方案”。

面试里你至少要做到：

1. 能解释什么是一致性，为什么分布式系统里更难。
2. 能回答接口重复提交怎么办。
3. 能回答消息重复消费怎么办。
4. 能回答为什么要用 Redis。
5. 能回答为什么要用消息队列。
6. 能回答高并发下数据库压力怎么处理。

一句话标准：

“我了解分布式系统里的典型问题，知道常见解决思路，也知道不同方案有代价和取舍。”

---

## 二、先建立一个整体认识

### 1. 什么叫分布式系统

一句话：

“原来一个系统跑在一台机器上，现在为了支撑更大流量、更高可用或更强扩展性，把它拆到多台机器上协同工作，这就是分布式系统。”

为什么要分布式：

1. 一台机器性能有限。
2. 单机故障风险太高。
3. 不同模块职责不同，拆开更容易扩展和维护。

为什么分布式更难：

1. 多台机器之间要通过网络通信，网络本身就不可靠。
2. 多副本之间可能数据不一致。
3. 服务拆分后，调用链更长，故障更容易扩散。
4. 会出现重复请求、消息重复、超时、重试、乱序等问题。

面试口语版：

“分布式的核心难点不是把程序拆开，而是拆开以后会引入网络不可靠、状态不一致、调用链变长、故障传播这些新问题，所以设计重点就从单机性能，变成了一致性、可用性、容错和治理能力。”

### 2. 什么叫中间件

一句话：

“中间件就是位于业务系统和底层资源之间，帮我们解决通用问题的一类基础组件。”

常见中间件：

1. `MySQL`：存业务数据。
2. `Redis`：做缓存，也可做计数、限流、分布式锁等。
3. `MQ`：做异步、解耦、削峰。
4. 注册中心、配置中心、网关：这些了解概念即可，不是你当前优先级。

## 三、分布式系统只背这些核心点

### 1. CAP

CAP 是什么：

1. `C`：一致性，读到的数据是不是同一份。
2. `A`：可用性，每次请求是不是都能得到响应。
3. `P`：分区容错，网络分区出现时系统还能不能继续工作。

怎么理解：

1. 分布式系统里，网络分区无法完全避免，所以 `P` 基本是必选项。
2. 真正需要权衡的是更偏 `C` 还是更偏 `A`。

面试回答：

“CAP 不是说三者只能选二，而是说在发生网络分区时，系统必须在一致性和可用性之间做取舍。分布式系统无法假设网络永远可靠，所以通常都要接受 P，然后按业务特点决定更偏强一致还是高可用。”

### 2. BASE

一句话：

“BASE 可以理解为对强一致性的放松，接受短时间不一致，但最终达到一致。”

包括：

1. Basically Available：基本可用。
2. Soft State：软状态，允许中间状态存在。
3. Eventually Consistent：最终一致。

适用场景：

1. 对实时强一致要求没那么高的业务。
2. 希望优先保证系统可用性和吞吐。

对比表：

| 概念 | 核心关注点 | 关键词 | 特点 | 常见取舍/场景 |
| --- | --- | --- | --- | --- |
| CAP | 分布式系统在网络分区下的取舍 | C、A、P | 发生网络分区时，一致性和可用性无法同时完全满足 | 面试里常说：分布式系统通常都要接受 P，再看更偏 C 还是 A |
| BASE | 对强一致性的放松 | Basically Available、Soft State、Eventually Consistent | 允许短时间不一致，但最终一致 | 电商、缓存、消息队列、推荐/搜索等更关注可用性和吞吐的场景 |
| 强一致性 | 写成功后立刻读到最新值 | 立即一致、实时一致 | 用户任何时刻读到的都是最新数据 | 银行转账、余额、库存扣减等对正确性要求极高的场景 |
| 最终一致性 | 不保证立刻一致，但保证最终一致 | 异步同步、短暂不一致、最终收敛 | 不同节点短时间看到的数据可能不同，过一段时间会一致 | 订单状态传播、缓存刷新、分布式副本同步 |

怎么记：

1. `CAP` 讲的是“分区出现时怎么取舍”。
2. `BASE` 讲的是“为了高可用，接受短暂不一致”。
3. 强一致性强调“现在必须一致”。
4. 最终一致性强调“可以晚一点，但最后要一致”。

### 3. 强一致性和最终一致性

强一致性：

“一旦写成功，后续读请求立刻都能读到最新数据。”

最终一致性：

“某个时刻可能暂时不一致，但经过一段时间后会达到一致。”

为什么分布式里更难：

1. 数据可能有多个副本。
2. 写请求需要同步到多台机器。
3. 网络延迟、超时、丢包都会影响同步结果。

面试常用答法：

“单机系统里只要一份数据，写完基本就结束了；分布式里往往有多副本和多节点，写入后还要同步、复制、确认，所以一致性就会被网络和节点故障放大，处理起来复杂很多。”

### 4. 服务拆分后为什么会有新问题

单体系统时：

1. 调用通常是本地函数调用。
2. 数据往往在一个库里。
3. 事务边界相对简单。

拆分成多个服务后：

1. 本地调用变成远程调用，可能超时、失败、重试。
2. 一个业务流程可能涉及多个服务和多个数据库。
3. 分布式事务、链路追踪、限流熔断、服务治理都会出现。

一句话：

“分布式不是简单地把服务拆开，而是把很多原本在单机内被隐藏的问题暴露出来。”

### 5. 分布式高频概念速记

#### 1. 分布式事务

一句话：

“一个业务操作要跨多个服务或多个数据库时，怎样保证整体结果正确。”

为什么会有：

1. 单体应用里通常可以用本地事务一起提交或回滚。
2. 分布式场景下，一个下单流程可能同时涉及订单、库存、账户等多个服务。
3. 这些服务的数据不在同一个库里，本地事务管不了全局。

常见思路：

1. `2PC`：两阶段提交，强调强一致，但性能和可用性压力较大。
2. `TCC`：Try、Confirm、Cancel，把业务拆成预留、确认、取消三个阶段。
3. `Saga`：长事务拆成多个本地事务，失败时走补偿。
4. 消息最终一致性：先做核心操作，再通过消息把后续动作异步补齐。

面试回答：

“分布式事务的本质是一个业务流程跨多个服务后，不能再直接依赖单库事务，所以要在一致性、可用性和复杂度之间做平衡。工程上常见做法有 2PC、TCC、Saga 和消息最终一致性，具体要看业务对一致性的要求有多高。”

#### 2. 链路追踪

一句话：

“把一次请求经过的所有服务调用串起来，方便定位问题和分析性能。”

为什么重要：

1. 微服务下一个请求通常会经过网关、订单、库存、支付等多个服务。
2. 只看单个服务日志，很难知道一次请求到底卡在哪一跳。
3. 出现慢请求、异常传播、超时时，需要完整调用链辅助排查。

核心关键词：

1. `TraceId`：一次请求的全局唯一标识。
2. `SpanId`：调用链中某一段操作的标识。
3. 父子调用关系：反映服务之间的调用层级。

面试回答：

“链路追踪主要解决微服务下排障困难的问题。它会给一次请求打上全局 TraceId，再把每一段调用记录成 Span，这样就能看到请求经过了哪些服务、哪一段最慢、哪一段报错。”

#### 3. 限流

一句话：

“限制单位时间内进入系统的请求量，防止突发流量把系统打垮。”

作用：

1. 保护应用自身。
2. 保护数据库、缓存、下游服务。
3. 在高并发时避免雪崩式故障。

常见算法：

1. 计数器。
2. 滑动窗口。
3. 令牌桶。
4. 漏桶。

面试回答：

“限流本质上是流量保护，核心目的是让系统在大流量下不要被瞬间压垮。常见做法有计数器、滑动窗口、令牌桶和漏桶，超出的请求可以选择排队、拒绝或降级处理。”

#### 4. 熔断

一句话：

“当下游服务持续失败或响应很慢时，暂时停止继续调用，防止故障扩散。”

为什么需要：

1. 下游故障时，如果上游还不断重试和等待，会把线程、连接等资源耗尽。
2. 一个服务出问题，可能拖垮整条调用链。

常见表现：

1. 错误率过高时快速失败。
2. 响应时间过长时直接熔断。
3. 熔断后过一段时间进入半开状态，探测服务是否恢复。

面试回答：

“熔断可以理解成保险丝机制。下游服务异常时，不再继续把请求打过去，而是先快速失败或走兜底逻辑，避免把上游也拖死。等下游恢复后，再逐步放开调用。”

#### 5. 服务治理

一句话：

“对大量微服务进行统一管理和协调的一整套能力。”

通常包括：

1. 服务注册与发现。
2. 负载均衡。
3. 配置中心。
4. 限流、熔断、降级。
5. 监控告警。
6. 链路追踪。
7. 灰度发布和健康检查。

面试回答：

“服务治理不是单个功能点，而是管理微服务体系的整体能力。它解决的是服务越来越多之后，怎么让服务能被发现、能稳定调用、能监控、能配置、能容灾的问题。”

怎么记：

1. 分布式事务：保证多个服务一起做事时结果正确。
2. 链路追踪：看清一次请求走过了哪些服务。
3. 限流：控制流量，避免系统被冲垮。
4. 熔断：下游故障时先断开，防止故障扩散。
5. 服务治理：把一堆微服务统一管起来。

### 6. 幂等

什么是幂等：

“同一个请求执行一次和执行多次，结果应该一样。”

为什么重要：

1. 前端可能重复提交。
2. 用户可能重复点击。
3. 网络超时后客户端会重试。
4. MQ 可能重复投递。

常见做法：

1. 业务唯一单号。
2. 去重表。
3. 唯一索引。
4. 状态机校验。
5. 幂等 token。

面试口语版：

“幂等本质上是防重复执行。比如订单创建、扣款、消息消费这些场景，都不能因为重试或重复投递导致业务被执行两次，常见方式是利用唯一业务键、数据库唯一约束、消费去重记录或者状态流转判断来保证。”

### 6. 重试

为什么要重试：

1. 有些失败是瞬时失败。
2. 网络抖动、临时超时并不代表业务一定失败。

重试的风险：

1. 可能把压力放大。
2. 可能造成重复操作。
3. 如果下游已经故障，盲目重试会雪上加霜。

正确理解：

“重试必须配合幂等、退避和上限，不能无脑重试。”

### 7. 去重

去重常见场景：

1. 接口重复提交。
2. MQ 重复消费。
3. 定时任务重复执行。

常见方案：

1. 唯一请求号。
2. 数据库唯一索引。
3. Redis 记录处理标记。
4. 消费记录表。

### 8. 分布式锁

它是干什么的：

“让多个节点在同一时刻只有一个节点能处理某个共享资源。”

适用场景：

1. 防止同一任务被多机器重复执行。
2. 控制某段关键逻辑只能有一个执行者。

这里的“同一资源”一般指：

1. 同一个订单。
2. 同一个任务。
3. 同一个用户的某次操作资格。
4. 同一个库存记录。
5. 同一个缓存重建动作。

一句话理解：

“只要多个节点同时处理它会产生冲突、重复或浪费，这个对象就可以看成同一资源。”

常见实现：

1. 基于 `Redis`。
2. 基于数据库。
3. 基于 `ZooKeeper`。

#### 1. 基于 Redis 的分布式锁

最常见，因为性能高，实现简单。

常见写法：

`SET lock_key unique_value NX EX 30`

怎么理解：

1. `lock_key`：表示要锁哪一个资源，比如 `lock:order:1001`。
2. `unique_value`：表示这把锁是谁加的，通常是随机值或请求唯一标识。
3. `NX`：只有 key 不存在时才允许设置。
4. `EX 30`：30 秒后自动过期。

为什么这样能当锁：

1. 如果 `lock_key` 不存在，设置成功，说明当前节点拿到了锁。
2. 如果 `lock_key` 已经存在，设置失败，说明锁已被别人持有。
3. 在 Redis 分布式锁里，key 本身就是锁的载体。

为什么释放锁时不能直接删：

1. 节点 A 拿到锁后业务执行很久。
2. 锁过期后，节点 B 又拿到了同一个锁。
3. 如果 A 结束时直接 `DEL key`，会把 B 的锁误删。

所以更安全的原则是：

1. 加锁时写入唯一 value。
2. 释放时先确认 value 还是自己的，再删除。

优点：

1. 实现简单。
2. 性能高。
3. 适合高并发短临界区场景。

风险：

1. 锁可能提前过期。
2. 可能误删别人的锁。
3. 如果锁粒度设计不合理，会影响吞吐。
4. 业务时间过长时处理比较麻烦。

#### 2. 基于数据库的分布式锁

常见思路：

1. 建一张锁表，通过插入唯一键来抢锁。
2. 或者利用数据库的悲观锁、乐观锁控制并发。

补充理解：

1. 悲观锁：假设冲突经常发生，所以先加锁再操作，典型如 `select ... for update`。
2. 乐观锁：假设冲突不常发生，通常不先加锁，而是在更新时通过 `version` 或时间戳校验数据有没有被别人改过。

一句话区分：

1. 悲观锁是“先锁住，再操作”。
2. 乐观锁是“先尝试操作，提交时再检查有没有冲突”。

崩溃后的差异也要知道：

1. 如果是“锁表 + 唯一键”方案，持锁节点崩溃后，锁记录可能残留，所以也要考虑过期时间、续约和超时清理。
2. 如果是数据库悲观锁，它通常和事务、连接绑定，进程崩溃或连接断开后，数据库一般会回滚事务并释放锁。
3. 如果是数据库乐观锁，它本质上不是长期持有锁，而是更新时做版本校验，所以一般没有“锁残留”问题。

怎么理解：

1. 抢锁成功，本质上是“某条唯一记录被你占到了”。
2. 其他节点再来插入同样的唯一键会失败。

优点：

1. 好理解。
2. 不依赖额外中间件时也能做。
3. 和业务数据放在同一个数据库里时，落地相对直接。

缺点：

1. 性能通常不如 Redis。
2. 会增加数据库压力。
3. 不适合高频激烈竞争场景。

#### 3. 基于 ZooKeeper 的分布式锁

核心思路：

1. 利用临时节点或有序节点实现抢锁。
2. 节点断开后，临时节点会自动消失。

常见做法：

1. 多个客户端都在同一个锁目录下创建临时有序节点。
2. 谁创建出来的节点编号最小，谁拿到锁。
3. 没拿到锁的客户端，不是一直轮询，而是监听自己前一个节点。
4. 当前一个节点被删除后，再判断自己是不是最小节点。
5. 如果自己已经是最小节点，就获得锁。

怎么理解：

1. `ZooKeeper` 本质上是让多个客户端在同一个目录下排队。
2. 最前面的节点拿到锁，后面的节点按顺序等待。
3. 如果持锁客户端宕机或会话断开，临时节点会自动删除，后面的客户端就有机会继续拿锁。

优点：

1. 一致性语义通常更强。
2. 更适合对锁可靠性要求较高的场景。

缺点：

1. 实现和运维成本更高。
2. 对你当前面试准备来说，不需要深挖实现细节。

#### 4. 常见问题

误删：

1. 你的锁过期了，别人拿到新锁。
2. 你执行结束后又把 key 删掉。
3. 结果把别人的锁删了。

死锁：

1. 如果锁没有过期时间，持锁节点挂了。
2. 这把锁可能永远不释放。

锁提前过期：

1. 你给锁设置了 30 秒。
2. 但业务执行了 50 秒。
3. 30 秒一到，Redis 会先删 key。
4. 其他节点又可以重新抢锁。

一句话：

“分布式锁真正难的地方不是加锁，而是锁的生命周期管理。”

#### 5. 业务执行时间过长怎么办

这是 Redis 分布式锁最常见的追问点。

问题本质：

1. `EX 30` 表示 key 30 秒后自动过期。
2. Redis 不知道你的业务有没有执行完。
3. 所以如果业务超过 30 秒，锁会先没掉，业务还在继续。

可能后果：

1. 原节点还在执行。
2. 新节点已经重新拿到锁。
3. 同一段逻辑被多个节点并发执行。

常见处理思路：

##### 方案一：把过期时间设得更合理

做法：

1. 让 TTL 明显大于业务正常执行时间。
2. 预留一定抖动余量。

优点：

1. 简单。
2. 成本低。

局限：

1. 只能降低风险，不能彻底解决。
2. 线上总会出现慢请求、卡顿、网络抖动。

##### 方案二：加续期机制

做法：

1. 先给锁一个初始过期时间。
2. 业务执行期间，后台定时检查锁是否还是自己的。
3. 如果锁还属于当前节点，并且业务还没结束，就继续延长过期时间。

怎么理解：

1. 这类机制常被叫做看门狗或自动续期。
2. 它的目标是“只要业务还正常跑，锁就不要提前过期”。

注意点：

1. 续期前也要先判断 value 是否还是自己的。
2. 不能把别人的锁续上。

##### 方案三：不要让锁包住整个长流程

这是更推荐的工程思路。

做法：

1. 锁只保护“抢执行权”这一小段。
2. 抢到后，立刻把任务状态改成 `PROCESSING` 之类的状态。
3. 然后尽快释放锁。
4. 后面的长流程靠状态机继续推进。

为什么更好：

1. 锁持有时间短，不容易超时。
2. 故障恢复更容易。
3. 不会把整个长任务都绑在一把锁上。

##### 方案四：把大任务拆小

做法：

1. 一个大任务拆成多个小批次或多个阶段。
2. 每次只锁住很短的一段动作。
3. 每一段完成后记录进度，再继续下一段。

好处：

1. 单次锁持有时间更短。
2. 更容易失败重试和断点恢复。

##### 方案五：用队列、状态机、唯一约束一起兜底

更稳妥的思路通常不是“只靠锁”，而是多种机制组合。

1. 分布式锁：负责抢执行权。
2. 状态机：负责控制流程能不能往下走。
3. 队列串行化：适合天然需要顺序执行的任务。
4. 数据库唯一约束：做最终防重复兜底。

一句话：

“锁是前置控制，不应该成为唯一的正确性保障。”

#### 6. 面试怎么答

参考回答：

“分布式锁是为了解决多节点竞争同一资源的问题。常见实现有 Redis、数据库和 ZooKeeper，其中 Redis 最常见，通常通过 `SET key value NX EX` 这种原子方式抢锁，并通过唯一 value 标识锁归属。它的难点不在能不能加上锁，而在过期时间、误删、锁提前失效和长流程处理。工程上常见做法是让锁只保护抢执行权这一小段，再结合状态机、队列或数据库唯一约束做兜底，而不是把整个长业务都压在一把锁上。”

你现在只需要会到的程度：

1. 知道它是为了解决多节点并发竞争。
2. 知道它不是万能的，设计不好会有误删、超时、死锁、锁提前过期等问题。
3. 知道长流程不能只靠一把简单分布式锁硬兜底。

### 9. Paxos / Raft 要掌握到什么程度

你不用学算法细节，只要记住：

1. 它们是为了解决分布式一致性问题的协议或算法思想。
2. 目标是让多个节点在面对故障时，对某个值或日志顺序达成一致。
3. `Raft` 相比 `Paxos` 更容易理解和工程实现。

简要理解 `Paxos`：

1. `Paxos` 更偏经典的一致性算法。
2. 它的核心思想是让一个提案被多数节点接受，从而形成一致结果。
3. 它关注的是“在多个节点可能故障、网络可能不稳定时，怎样仍然选出一个被多数认可的值”。

它大概怎么运作：

1. 某个节点先提出一个带编号的提案。
2. 它先去问多数节点，能不能先承诺不再接受比这个编号更小的旧提案。
3. 如果拿到了多数节点的承诺，它再正式发起“接受这个值”的请求。
4. 如果最终有多数节点接受了这个值，这个值就可以认为达成了一致。

怎么记：

1. `Paxos` 关键不是“随便投票”，而是“提案编号 + 两阶段 + 多数派接受”。
2. 它的目的是在并发提议、网络延迟、节点故障的情况下，最终只收敛出一个被多数认可的结果。

简要理解 `Raft`：

1. `Raft` 也是为了解决一致性问题。
2. 它更强调先选出 `leader`，再由 `leader` 统一做日志复制和提交。
3. 它把一致性问题拆成了选主、日志复制、故障恢复几个更容易理解的部分。

一句话区别：

1. `Paxos` 更偏“多数派接受同一个提案”。
2. `Raft` 更偏“先选 leader，再由 leader 带大家保持一致”。

面试说法：

“Paxos 和 Raft 我目前理解到用途层面，它们都是分布式一致性相关的协议，核心目标是在多节点环境下就某个状态或日志顺序达成一致。工程上 Raft 通常更容易理解和实现。”

## 四、常用中间件要掌握到什么程度

### 1. MySQL

你已经有单独的 MySQL 文档，这里只记和分布式面试强相关的部分。

必须会答：

1. 索引为什么快，为什么是 `B+` 树。
2. 事务隔离级别和 `MVCC` 是什么。
3. 慢 SQL 怎么优化。
4. 高并发下数据库压力怎么处理。

高并发下数据库压力怎么处理：

1. 加缓存，把热点读挡在 `Redis`。
2. 优化 SQL 和索引。
3. 读写分离。
4. 分库分表。
5. 批量处理，减少频繁小事务。

一句话：

“数据库是最贵的核心资源，要优先通过索引、缓存、批量化、读写分离等手段减压。”

### 2. Redis

### 为什么用 Redis

一句话：

“因为 Redis 基于内存，读写很快，适合做高频访问数据的缓存。”

常见用途：

1. 缓存热点数据。
2. 计数器。
3. 会话信息。
4. 分布式锁。
5. 排行榜。
6. 限流。

### Redis 常见数据结构

先记一个总原则：

1. Redis 本质上还是 `key-value` 存储。
2. `key` 一般是字符串形式的标识。
3. 真正变化丰富的是 `value`，它可以是不同数据结构。

常见理解：

1. `key` 像名字，比如 `user:1001`、`order:2002`、`rank:game`。
2. `value` 才可能是 `String`、`Hash`、`List`、`Set`、`ZSet` 这些结构。

| 类型 | `key` 示例 | `value` 结构 | 适合场景 | 常用操作 |
| --- | --- | --- | --- | --- |
| `String` | `token:user:1001` | 单个字符串、数字、序列化对象 | 简单缓存、计数器、分布式锁 | `SET` `GET` `INCR` `DECR` `SETNX` `EXPIRE` |
| `Hash` | `user:1001` | 一个对象的多个字段 | 用户信息、商品信息、配置对象 | `HSET` `HGET` `HMGET` `HGETALL` `HDEL` |
| `List` | `comments:post:88` | 按顺序排列的一串元素 | 简单消息队列、评论列表、动态流 | `LPUSH` `RPUSH` `LPOP` `RPOP` `LRANGE` |
| `Set` | `tags:article:10` | 无序去重集合 | 标签、去重、共同好友、黑白名单 | `SADD` `SMEMBERS` `SISMEMBER` `SREM` `SUNION` `SINTER` |
| `ZSet` | `rank:game:s1` | 带 `score` 的有序集合 | 排行榜、延迟任务、按分数/时间排序 | `ZADD` `ZRANGE` `ZREVRANGE` `ZSCORE` `ZREM` `ZRANK` |

一句话速记：

1. `String`：最通用，简单值、计数、锁。
2. `Hash`：存对象字段。
3. `List`：按顺序存一串数据。
4. `Set`：无序去重集合。
5. `ZSet`：带分数的有序集合，最适合排行榜。

### Redis 常用操作语法速记

#### 1. String 常用操作

```text
SET key value
# 设置一个字符串值

GET key
# 读取 key 对应的值

INCR key
# 对数字值加 1，常用于计数器

DECR key
# 对数字值减 1

SETNX key value
# 只有 key 不存在时才设置成功，常用于抢锁

EXPIRE key seconds
# 给 key 设置过期时间，单位是秒
```

#### 2. Hash 常用操作

```text
HSET key field value
# 给 hash 中某个字段赋值

HGET key field
# 读取某个字段的值

HMGET key field1 field2
# 一次读取多个字段

HGETALL key
# 读取这个 hash 的所有字段和值

HDEL key field
# 删除某个字段
```

#### 3. List 常用操作

```text
LPUSH key value
# 从左边插入一个元素

RPUSH key value
# 从右边插入一个元素

LPOP key
# 从左边弹出一个元素

RPOP key
# 从右边弹出一个元素

LRANGE key start end
# 按下标范围读取元素，比如 0 到 9 表示前 10 个
```

#### 4. Set 常用操作

```text
SADD key member
# 向集合中添加元素

SMEMBERS key
# 查看集合中所有元素

SISMEMBER key member
# 判断某个元素是否在集合中

SREM key member
# 删除集合中的某个元素

SUNION key1 key2
# 求两个集合的并集

SINTER key1 key2
# 求两个集合的交集
```

#### 5. ZSet 常用操作

```text
ZADD key score member
# 添加一个带分数的元素

ZRANGE key start end
# 按分数从小到大读取某个排名范围

ZREVRANGE key start end
# 按分数从大到小读取某个排名范围

ZSCORE key member
# 查看某个元素的分数

ZREM key member
# 删除某个元素

ZRANK key member
# 查看某个元素当前排名
```

### Redis 支持哪些查询

先说结论：

1. Redis 支持的是“按结构设计好的访问方式”去查。
2. 它不擅长像 MySQL 一样做通用条件查询。
3. 所以 Redis 更像“高性能数据结构服务器”，不是“通用查询数据库”。

常见支持的查询方式：

1. 按 `key` 精准查。
2. 查某个 `Hash` 里的指定字段。
3. 按 `List` 下标范围取元素。
4. 对 `Set` 做成员判断、交集、并集、差集。
5. 对 `ZSet` 按排名范围或按 `score` 范围查询。

可以这样理解：

1. `String`：擅长按 `key` 精准取值，不擅长复杂筛选。
2. `Hash`：擅长读取某个对象的字段，不擅长跨很多对象按字段过滤。
3. `List`：擅长按位置范围取数据，比如前 10 条。
4. `Set`：擅长集合运算，比如共同好友、共同标签。
5. `ZSet`：最适合做范围查询，比如排行榜前 100、某个分数区间。

Redis 不擅长的查询：

1. 不能像 MySQL 一样天然支持 `where age > 20 and city = 'Shanghai'` 这类组合条件查询。
2. 不适合临时按任意字段筛选、排序、分页。
3. 如果要支持复杂查询，通常要提前设计好多组 `key` / `Set` / `ZSet` 作为访问路径。

一句话：

“Redis 支持按 key、按字段、按位置、按集合、按分数范围去查，但不擅长关系型数据库那种灵活的条件组合查询。”

### Redis 为什么快

常见回答：

1. 数据放在内存里。
2. 数据结构设计高效。
3. 单线程模型避免了大量线程切换和锁竞争。
4. 网络 IO 模型高效。

注意：

“单线程”不是说 Redis 只能处理很少请求，而是说它把核心命令执行做成了串行，避免复杂锁竞争；真正快的根本原因还是内存访问快和实现高效。

### 缓存穿透、击穿、雪崩

缓存穿透：

“查询一个数据库和缓存里都不存在的数据，导致请求每次都打到数据库。”

常见处理：

1. 参数校验。
2. 布隆过滤器。
3. 空值缓存。

缓存击穿：

“某个热点 key 失效瞬间，大量请求同时打到数据库。”

常见处理：

1. 热点数据不过期或逻辑过期。
2. 加互斥锁。
3. 提前刷新。

补充几个高频追问：

#### 1. 为什么常说“逻辑过期 + 互斥锁”是好组合

怎么理解：

1. 逻辑过期：缓存数据本身先不立刻删掉，而是在数据里额外带一个“过期时间”字段。
2. 即使这个时间到了，旧数据还在，可以先返回旧值，避免请求直接全部打到数据库。
3. 互斥锁：只允许一个线程去做缓存重建，其他线程不要同时回源。

为什么这两个搭配好：

1. 逻辑过期解决的是“缓存不能一下子消失”，先保证可用。
2. 互斥锁解决的是“只能有一个线程去刷新”，避免并发打库。
3. 所以它兼顾了可用性和数据库保护。

一句话：

“逻辑过期负责先顶住流量，互斥锁负责控制重建并发。”

#### 2. 异步刷新失败怎么办

常见思路：

1. 刷新失败时不要立刻把旧数据删掉。
2. 记录日志和监控，方便发现问题。
3. 下次请求到来时，继续尝试触发刷新。
4. 必要时加重试机制或降级兜底。

核心原则：

1. 宁可短时间返回旧数据，也不要让缓存直接空掉把压力打到数据库。

#### 3. 如何避免多个线程同时刷新

常见做法：

1. 发现数据逻辑过期后，先尝试抢互斥锁。
2. 抢到锁的线程负责回源数据库并重建缓存。
3. 没抢到锁的线程直接返回旧值，或者稍后重试。

双重检查锁思想：

1. 线程拿到锁后，不要立刻查库。
2. 先再检查一次缓存，看是不是已经被别的线程刷新好了。
3. 如果已经刷新好了，就不用重复查数据库。

一句话：

“双重检查锁就是抢到锁后再看一眼，避免重复刷新。”

缓存雪崩：

“大量 key 在同一时间失效，导致请求集中冲击数据库。”

常见处理：

1. 过期时间加随机值。
2. 多级缓存。
3. 限流降级。
4. 缓存预热。

一段顺口答案：

“缓存穿透是查不存在的数据，击穿是单个热点 key 过期，雪崩是大量 key 同时失效。对应思路一般分别是空值缓存或布隆过滤器、热点保护和互斥控制、过期时间打散加限流降级。”

### 3. MQ

### 为什么要用消息队列

三大关键词：

1. 解耦。
2. 削峰。
3. 异步。

怎么理解：

1. 解耦：上游不需要直接依赖下游所有处理逻辑。
2. 削峰：瞬时高流量先写入队列，后面慢慢消费。
3. 异步：非核心链路不阻塞主流程。

面试回答：

“消息队列的核心价值是解耦、削峰和异步。比如主流程先把核心动作做完，再把非强实时操作通过 MQ 异步下发，可以缩短接口响应时间，也能避免流量高峰直接压垮下游。”

### MQ 会带来什么问题

1. 消息丢失。
2. 重复消费。
3. 消息积压。
4. 顺序问题。
5. 下游失败后的重试问题。

### 消息重复消费怎么办

标准答案：

1. 不要指望 MQ 绝对不重复。
2. 消费端必须做幂等。
3. 常见方法是业务唯一键、去重表、状态判断、唯一索引。

一段口语回答：

“消息系统通常更强调至少投递一次，所以重复消费是要靠消费端幂等来兜底的。常见做法是给消息绑定业务唯一标识，在消费时先查是否处理过，或者直接依赖数据库唯一约束和状态机来保证重复执行不会产生副作用。”

### 消息丢失怎么理解

面试里不用讲太深，只要知道：

1. 生产者发送时可能丢。
2. MQ 存储时可能丢。
3. 消费者处理后确认前可能丢。

答法：

“消息丢失要分发送端、Broker 和消费端三个阶段去看，一般通过发送确认、持久化、消费确认和失败重试来降低风险，但工程上通常追求的是可接受风险，而不是绝对零丢失。”

### 消息积压怎么办

常见思路：

1. 先看是生产太快还是消费太慢。
2. 扩容消费者。
3. 提高单条消费效率。
4. 降级非核心业务。
5. 必要时做限流。

## 五、面试最常问的 8 个问题

### 1. 什么是一致性，为什么分布式系统里更难

参考回答：

“一致性可以理解为多个节点看到的数据是否一致。单机系统通常只维护一份状态，写入后问题相对简单；分布式系统里往往有多个副本和多个节点，写入后还需要跨网络同步。由于网络可能延迟、超时、丢包，节点也可能故障，所以一致性问题在分布式环境里会被显著放大。”

### 2. CAP 怎么理解

参考回答：

“CAP 指的是一致性、可用性和分区容错。分布式系统里网络分区不可完全避免，所以通常都要接受 P；真正的取舍是在分区发生时，系统更偏向保证一致性还是保证可用性。”

### 3. 什么是幂等

参考回答：

“幂等就是同一个请求执行一次和执行多次，对系统产生的结果是一样的。它主要是为了解决重试、重复提交和重复消费带来的副作用问题。”

### 4. 接口重复提交怎么办

参考回答：

“核心是保证幂等。常见做法包括前端防重复点击、服务端生成幂等号、数据库唯一索引、状态机校验等。真正兜底的一般还是服务端的业务唯一约束，而不是只靠前端控制。”

### 5. 为什么要用 Redis

参考回答：

“Redis 最常见的作用是做缓存，因为它基于内存、访问速度快，能把大量热点读请求挡在数据库前面，从而降低数据库压力。同时它也支持计数器、分布式锁、排行榜、限流等能力。”

### 6. Redis 为什么快

参考回答：

“核心原因是数据放在内存中，访问成本很低；同时 Redis 本身数据结构高效，命令执行路径短，单线程模型也减少了锁竞争和线程切换开销，所以整体吞吐很高。”

### 7. 为什么要用消息队列

参考回答：

“消息队列主要解决解耦、削峰和异步。业务主流程可以先完成核心动作，把非核心处理交给下游异步消费，这样既能缩短响应时间，也能缓冲高峰流量对下游系统的冲击。”

### 8. 消息重复消费怎么办

参考回答：

“重复消费通常靠消费端幂等来处理，而不是假设 MQ 一定只投递一次。一般会结合业务唯一键、去重记录表、唯一索引和状态流转判断，保证同一条消息多次处理也不会产生重复副作用。”

## 六、你现在最该背的最小知识集

如果时间特别紧，只看下面这些：

1. 分布式系统更难，因为网络不可靠、节点会故障、数据可能不一致。
2. `CAP` 是一致性、可用性、分区容错；分区时要做取舍。
3. 强一致性要求马上读到最新值，最终一致性允许短暂不一致。
4. 幂等是防止重复执行。
5. 重试必须配合幂等，不能无脑重试。
6. `Redis` 主要用来做缓存，快是因为内存和高效实现。
7. 缓存常见问题是穿透、击穿、雪崩。
8. `MQ` 的核心价值是解耦、削峰、异步。
9. 消息重复消费靠消费端幂等处理。
10. 高并发下数据库减压常见方法是缓存、索引优化、读写分离、分库分表。

---

## 七、你需要主动规避的风险

1. 不要装作自己做过复杂分布式架构。
2. 不要一上来就讲很深的协议细节。
3. 不要把“最终一致性”说成“数据永远可以不一致”。
4. 不要把“加了 Redis”说成万能优化，要承认会带来一致性和缓存失效问题。
5. 不要把“用了 MQ”说成天然可靠，要知道它会带来重复、积压、顺序等新问题。

更稳妥的表达方式：

“这部分我主要掌握到常见问题和通用处理思路的层面，如果结合具体组件实现细节，我目前了解的是工程常见做法。”

## 八、最后的面试策略

你的目标不是证明自己是资深分布式专家，而是证明：

1. 你知道这些问题是什么。
2. 你知道为什么会出现。
3. 你知道常见处理方案。
4. 你知道方案有代价和边界。
5. 你虽然不是这块出身，但学习和落地能力没问题。

如果面试官继续深挖，你可以这样收：

“这块我之前不是主做方向，所以我现在重点掌握的是高频概念、典型问题和工程上常见的解决思路。如果在实际项目里落地，我会再结合具体组件和业务一致性要求做更细设计。”
