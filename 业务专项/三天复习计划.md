# 业务专项三天复习计划

## 一、复习目标

结合你的简历和岗位要求，这个岗位大概率重点看以下几类能力：

1. C/C++ 基础是否扎实，能否胜任交易系统核心模块开发。
2. Linux 下多线程、网络编程、IO 多路复用是否能讲透。
3. 是否真正做过高并发优化，能把项目中的性能收益讲明白。
4. 是否理解证券交易系统/接入网关类业务的基本链路和设计重点。
5. 是否具备分布式系统基础，至少能回答一致性、缓存、消息、数据库相关问题。

你的优势很明确：

1. 有真实的 C++ 核心模块开发经验。
2. 有性能优化成果，且有量化指标，简历亮点足够。
3. 做过接入网关、规则校验、流量监控、信创适配，这些都很贴近岗位。

你三天内最该补的不是“再学一堆新东西”，而是：

1. 把已有项目经历包装成可追问、可展开、可落到原理的回答。
2. 把 Linux 网络编程、IO 多路复用、多线程同步这些底层知识补成体系。
3. 把分布式基础和证券业务常见问法补齐到“能答七八成”的程度。

## 二、三天策略

总体原则：

1. 以面试产出为导向，不做大而全复习。
2. 每天都要同时覆盖“八股 + 项目表达 + 手写/思考题”。
3. 每天结束前都要产出自己的口述答案，而不只是看资料。

建议每天投入 10 到 12 小时，按照“上午硬知识、下午项目与业务、晚上输出与模拟”推进。

## Day 1：补齐 Linux 网络编程主线，先把底层讲顺

### 上午：IO 多路复用 + Socket 基础

复习重点：

1. `select` / `poll` / `epoll` 的区别。
2. `epoll` 的 LT / ET、惊群、水平触发与边沿触发。
3. 阻塞 / 非阻塞 socket。
4. TCP 三次握手、四次挥手、TIME_WAIT、粘包拆包。
5. Reactor 模型、单 Reactor 单线程/多线程。

必须达到的效果：

1. 能解释为什么 `epoll` 更适合高并发连接。
2. 能说清一个请求从 accept 到 read/write 的处理流程。
3. 能回答“为什么 ET 模式下必须循环读到 `EAGAIN`”。

建议结合你的项目去串：

1. 统一接入产品本质上就是网关型系统，可以把“连接管理、协议解析、规则校验、转发”作为主链路讲述。
2. 流量监控模块可以顺带类比“异步拉取 + 多线程解析 + 单线程落库”的流水线设计思想。

### 下午：多线程、内存与性能优化

复习重点：

1. 线程和进程区别，线程安全问题。
2. 互斥锁、自旋锁、读写锁、条件变量、原子变量适用场景。
3. 生产者消费者模型、线程池设计。
4. 内存池、对象复用、减少拷贝、批量提交、背压控制。
5. 常见性能瓶颈定位思路：CPU、锁竞争、内存分配、系统调用、磁盘/网络。

必须达到的效果：

1. 能讲清你流量监控模块里的三阶段异步流水线为什么合理。
2. 能讲清固定块内存池优化的收益点。
3. 能回答“为什么不用无锁队列/为什么这里用多线程而不是多进程”。

这部分要重点准备你的项目话术：

1. 业务背景是什么。
2. 当时的瓶颈是什么。
3. 你如何定位问题。
4. 方案为什么这样设计。
5. 指标提升多少。
6. 代价和边界是什么。

### 晚上：项目一轮深挖

只做两件事：

1. 写出两个项目的 STAR 版本答案。
2. 为每个项目补 10 个追问题并自己口述。

建议优先准备这两个项目：

1. 规则校验性能重构。
2. 流量监控模块设计与优化。

今晚产出物：

1. 每个项目 3 分钟介绍稿。
2. 每个项目 10 个高频追问题。
3. 每个项目 1 个“如果重做一次你会怎么优化”的总结。

## Day 2：补齐数据库、分布式和证券业务理解

### 上午：数据库与中间件基础

复习重点：

1. MySQL 索引、B+ 树、覆盖索引、回表、最左匹配。
2. 事务四大特性、隔离级别、MVCC、幻读。
3. 常见 SQL 优化思路。
4. Redis 基础数据结构、缓存穿透/击穿/雪崩。
5. 消息队列的作用：解耦、削峰、异步。

必须达到的效果：

1. 能回答数据库为什么选这种结构。
2. 能回答高并发场景下数据库压力怎么处理。
3. 能联系你做过的 OceanBase / GoldenDB 适配经验，说出迁移中常见关注点。

结合简历重点准备：

1. 为什么信创迁移难。
2. 国产数据库适配通常会遇到哪些兼容性问题。
3. SQL、驱动、语法、事务、性能、部署环境分别可能出什么问题。

### 下午：分布式系统基础

岗位要求里这一块是你的短期补强重点，建议重点掌握“能解释、能举例”的程度。

复习重点：

1. CAP、BASE 基本理解。
2. 一致性问题：强一致性、最终一致性。
3. 分布式锁、幂等、重试、去重。
4. 服务拆分后为什么会带来新问题。
5. 常见协议或思想：Paxos/Raft 了解到用途和核心目标即可。

必须达到的效果：

1. 能回答“什么是一致性，为什么分布式系统里更难”。
2. 能回答“接口重复提交怎么办”。
3. 能回答“消息重复消费怎么办”。
4. 能回答“交易系统为什么更强调可靠性和时延稳定性”。

这里不用死磕算法细节，三天时间不值得。重点是建立一套面试可用表达：

1. 先定义问题。
2. 再说常见方案。
3. 最后说权衡。

### 晚上：证券交易系统和接入网关专项准备

这是你最容易拉开差距的一晚。

复习重点：

1. 接入网关在交易系统中的位置。
2. 行情、交易、资金、账户几个核心域的基本关系。
3. 为什么接入层要做校验、限流、协议转换、路由转发。
4. 交易场景为何强调低延迟、稳定性、可追踪、灰度和回滚能力。
5. 线上问题排查思路：日志、监控、链路、指标、压测。

建议准备的高频问题：

1. 你做的统一接入产品解决了什么问题。
2. 为什么要做规则校验，规则变多后性能为什么下降。
3. TPS 损耗从 2000 到 50 是怎么测出来的。
4. 如果线上延迟突然升高，你怎么排查。
5. 如果某券商迁移信创环境后性能下降，你先看哪里。

今晚产出物：

1. 一份“统一接入产品架构图”的文字版。
2. 一份“线上性能问题排查 checklist”。
3. 一份“信创迁移适配经验总结”。

## Day 3：强化表达、查漏补缺、模拟面试

### 上午：C++ 高频面试题冲刺

复习重点：

1. 左值右值、移动语义、完美转发。
2. 智能指针原理与循环引用。
3. 虚函数、虚表、继承、多态。
4. `new/delete` 与 `malloc/free` 区别。
5. STL 常见容器底层：`vector`、`map`、`unordered_map`。
6. `map` 和 `unordered_map` 使用场景。
7. RAII、异常安全、资源管理。

必须达到的效果：

1. 回答不只停留在定义，要能说到适用场景。
2. 能把你在项目里如何用智能指针、如何控制资源释放讲出来。

你的 `简历.md` 里已经提到线程异常、`unique_ptr/shared_ptr + deleter`、内存池回收，这部分很适合拿来做加分回答。

### 下午：全真模拟面试

按下面顺序至少完整模拟 2 轮：

1. 自我介绍 3 分钟。
2. 项目深挖 20 分钟。
3. Linux/C++ 基础 20 分钟。
4. 数据库/分布式 15 分钟。
5. 反问面试官 5 分钟。

模拟要求：

1. 所有回答尽量控制在 1 到 2 分钟内。
2. 先说结论，再说原因，再补细节。
3. 项目题一定落到“你的动作”和“量化结果”。

建议重点修正的问题：

1. 回答太散，没有结论先行。
2. 一上来就讲细节，没有先交代业务背景。
3. 能说做了什么，但说不清为什么这么设计。
4. 指标提升很亮眼，但没有解释测试口径。

### 晚上：最后收口，只看错题和高频题

不要再开新专题。

只做以下三件事：

1. 过一遍这三天记录的错题和不会答的问题。
2. 重复口述两遍自我介绍和两个核心项目。
3. 整理出明天面试前 30 分钟可快速浏览的一页纸。

这张一页纸建议只保留：

1. 自我介绍关键词。
2. 两个项目的业务背景、难点、方案、收益。
3. `epoll`、线程同步、内存池、Redis、MySQL、CAP 的一句话答案。
4. 2 个你最想主动展示的亮点。

1. 智能指针原理。

3. 线程池怎么设计。

## 四、针对你简历的面试表达建议

### 自我介绍主线

建议你把自己定位成：

“有证券交易接入系统经验的 C++ 工程师，做过高并发核心模块开发、性能优化、信创适配与流量监控模块设计，对 Linux 多线程、网络编程和服务性能优化有实际项目经验。”

### 两个核心卖点

1. 不是只会写业务逻辑，而是做过核心链路性能优化，并且有明确指标。
2. 不只是写单点功能，还参与过接入网关、信创迁移、跨端配套维护，说明协作和系统视角不错。

### 需要主动规避的风险

1. 不要把 `epoll` 只写在“了解”层面，面试时要尽量答到工程细节。
2. 分布式部分如果不熟，不要硬装很深；答到概念、问题、常见方案和权衡即可。
3. 指标类成果要能说明测试场景，否则容易被追问。
