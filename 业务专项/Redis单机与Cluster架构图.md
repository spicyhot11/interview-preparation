# Redis 单机与 Cluster 架构图

这份文档只用代码块做最直观的结构示意，方便快速理解。

---

## 1. Redis 单机架构

```text
                +-------------------+
                |    Client / App   |
                +---------+---------+
                          |
                          | read / write
                          v
                +-------------------+
                |   Redis Instance  |
                |   单机 / 单主节点   |
                +---------+---------+
                          |
                          | miss
                          v
                +-------------------+
                |    MySQL / DB     |
                +-------------------+
```

怎么理解：

1. 所有请求都打到同一个 Redis 实例。
2. 这个实例负责全部 key 的读写。
3. 如果缓存 miss，再去访问数据库。

特点：

1. 结构最简单。
2. 部署成本低。
3. 容量和性能都受单机限制。
4. 单点故障风险高。

---

## 2. Redis 主从架构

```text
                     +-------------------+
                     |    Client / App   |
                     +---------+---------+
                               |
                 +-------------+-------------+
                 |                           |
                 | write                     | read
                 v                           v
       +-------------------+       +-------------------+
       |   Redis Master    | ----> |   Redis Replica   |
       |       主节点       | sync  |       从节点       |
       +-------------------+       +-------------------+
                 |
                 | sync
                 v
       +-------------------+
       |   Redis Replica   |
       |       从节点       |
       +-------------------+
```

怎么理解：

1. 写请求通常进主节点。
2. 主节点把数据同步给从节点。
3. 读请求可以分摊到从节点。

特点：

1. 读能力增强。
2. 仍然不是严格意义上的分片集群。
3. 主节点压力仍然集中。

---

## 3. Redis Cluster 架构

```text
                         +-------------------+
                         |    Client / App   |
                         +---------+---------+
                                   |
                                   | key -> slot
                                   v
      +-------------------+   +-------------------+   +-------------------+
      |   Master Node A   |   |   Master Node B   |   |   Master Node C   |
      |   slots 0~5460    |   | slots 5461~10922  |   | slots 10923~16383 |
      +---------+---------+   +---------+---------+   +---------+---------+
                |                         |                         |
                | replica                 | replica                 | replica
                v                         v                         v
      +-------------------+   +-------------------+   +-------------------+
      |    Replica A1     |   |    Replica B1     |   |    Replica C1     |
      +-------------------+   +-------------------+   +-------------------+
```

怎么理解：

1. Redis Cluster 会把全部 key 映射到 `16384` 个槽位。
2. 每个主节点负责一部分槽位。
3. 请求会根据 key 所在槽位路由到对应主节点。
4. 每个主节点通常还会挂自己的从节点做高可用。

特点：

1. 不同 key 分散在不同主节点上。
2. 既能扩容容量，也能分散读写压力。
3. 某个主节点故障时，可以由对应副本提升。

---

## 4. 单机和 Cluster 的核心区别

```text
单机:
  所有 key -> 同一个 Redis 实例

Cluster:
  不同 key -> 先算 slot -> 再路由到不同 Master
```

再直观点：

```text
单机:
  user:1     -> Redis-1
  order:10   -> Redis-1
  rank:game  -> Redis-1

Cluster:
  user:1     -> slot X -> Master A
  order:10   -> slot Y -> Master B
  rank:game  -> slot Z -> Master C
```

---

## 5. 面试里可以怎么说

```text
Redis 单机架构就是所有请求都落到一个实例上，结构简单，但容量、
性能和可用性都受单机限制。

Redis Cluster 会把 key 映射到 16384 个槽位，再把槽位分配到多个
Master 节点上，每个 Master 还可以有 Replica 做高可用，所以它既
解决了单机容量瓶颈，也能分散读写压力。
```
